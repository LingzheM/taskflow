# TaskFlow

A real-time collaborative Kanban board. Full-stack TypeScript monorepo.

---

## Quick Start

### Option A — Docker (recommended)

```bash
cp .env.example .env        # adjust values if needed
docker compose up --build
```

| Service  | URL                    |
|----------|------------------------|
| Frontend | http://localhost:5173  |
| Backend  | http://localhost:4000  |

### Option B — Local dev

Prerequisites: Node 18+, pnpm 8+, PostgreSQL 16.

```bash
cp .env.example .env        # set DATABASE_URL to your local PG instance
pnpm install
pnpm --filter @taskflow/shared build   # compile shared types once
pnpm --filter @taskflow/backend exec prisma migrate dev
pnpm --filter @taskflow/backend exec prisma db seed  # optional seed data
```

Then in two terminals:

```bash
# terminal 1 — backend (port 4000)
pnpm --filter @taskflow/backend dev

# terminal 2 — frontend (port 5173)
pnpm --filter @taskflow/frontend dev
```

## Architecture

```
┌──────────────────────────────────────────────────────┐
│                     Browser                          │
│  React 18 + Zustand + TanStack Query + dnd-kit       │
│                        │  REST + WebSocket            │
└────────────────────────┼─────────────────────────────┘
                         │
┌────────────────────────▼─────────────────────────────┐
│              Hono (Node.js 18)                        │
│  Auth · Board · Column · Card routers                 │
│  Socket.io — room-per-board broadcast                 │
│                        │  Prisma ORM                  │
└────────────────────────┼─────────────────────────────┘
                         │
┌────────────────────────▼─────────────────────────────┐
│                  PostgreSQL 16                        │
│  Users · Boards · Members · Columns · Cards          │
│  Activity Log                                         │
└──────────────────────────────────────────────────────┘
```

Data flow for a drag operation:

```
User drags card
  → dnd-kit fires onDragEnd
    → Zustand store updated immediately (optimistic)
    → PATCH /api/cards/:id/move called in background
      → success: done
      → error:   Zustand rolls back to pre-drag snapshot
                 + toast shown to user
```

Monorepo layout:

```
taskflow/
├── apps/
│   ├── backend/   Hono + Prisma + Socket.io
│   └── frontend/  React + Vite + Tailwind
└── packages/
    └── shared/    TypeScript types shared across apps
```

## Tech Decisions

| Area | Choice | Reason |
|------|--------|--------|
| **Backend framework** | Hono | Lightweight, first-class TypeScript, ~10× faster than Express in benchmarks, same middleware pattern |
| **ORM** | Prisma | Type-safe queries, migration tooling, great DX |
| **State (server)** | TanStack Query | Automatic caching, background refetch, stale-while-revalidate — reduces boilerplate vs manual `useEffect` fetching |
| **State (client)** | Zustand | Minimal API, no providers, easy to write snapshot-based rollback for optimistic updates |
| **Drag-and-drop** | dnd-kit | Headless, accessible, composable sensors, first-class TypeScript support |
| **Monorepo** | pnpm workspaces | `workspace:*` protocol for shared package without publishing |

## Sorting Scheme — Fractional Indexing

### The Problem

Every Column and Card needs a persistent sort order. The naive approach — integers `1, 2, 3, ...` — requires updating O(N) rows when an item is inserted or moved (shift everything after the target). With 1 000 cards, moving the last card to position 1 rewrites all 1 000 rows.

### The Solution

We store `position` as a **lexicographically-sorted string** generated by the [`fractional-indexing`](https://www.npmjs.com/package/fractional-indexing) library (Figma's algorithm).

```
Before move:          After inserting D between A and B:
  A  →  "a0"            A  →  "a0"
  B  →  "a1"            D  →  "a0V"   ← new key, between "a0" and "a1"
  C  →  "a2"            B  →  "a1"
                         C  →  "a2"
```

**Why strings instead of floats?**

Floating-point numbers lose precision after ~50 bisections (you eventually can't fit a value between two adjacent floats). Fractional indexing encodes the same concept in base-95 strings that grow slowly and are safe to store and compare in any database.

### API Impact

| Operation | DB writes |
|-----------|-----------|
| Add card to end of column | 1 |
| Reorder card within column | 1 (just the moved card) |
| Move card to different column | 1 |
| Reorder columns | N (bulk transaction — one write per column) |

Column reorder writes all N columns because the client sends the full new ordering after an `arrayMove`. This could be optimised to write only the moved column (computing its new position between its new neighbours), but the current approach is simpler and N columns per board is small in practice.

### Database Query

```sql
-- columns and cards are always loaded pre-sorted by the API
SELECT * FROM columns WHERE board_id = $1 ORDER BY position ASC;
SELECT * FROM cards   WHERE column_id = $1 ORDER BY position ASC;
```

No extra application-side sort needed — the database index on `position` handles it.

### Known Limitation — Concurrent Moves

If two users drag the same card simultaneously, the last write wins (no version field). For a Kanban board with small teams this is acceptable. A proper fix would add an `updatedAt` optimistic lock: reject the write if the server's `updatedAt` is newer than the client's.

## API Reference

```
POST   /api/auth/register
POST   /api/auth/login

GET    /api/boards
POST   /api/boards
GET    /api/boards/:id           ← returns board + columns + cards (sorted)
DELETE /api/boards/:id
POST   /api/boards/:id/members

POST   /api/boards/:boardId/columns
PATCH  /api/columns/:id
DELETE /api/columns/:id
PATCH  /api/boards/:boardId/columns/reorder

POST   /api/columns/:columnId/cards
PATCH  /api/cards/:id
DELETE /api/cards/:id
PATCH  /api/cards/:id/move
```

All endpoints return:

```json
{ "success": true,  "data": <payload> }
{ "success": false, "error": { "code": "NOT_FOUND", "message": "..." } }
```

## Known Issues & Improvements

| Issue | Notes |
|-------|-------|
| No refresh-token rotation | JWT expires, user must log in again |
| Column reorder writes all N rows | Could write only the moved column |
| No virtual scrolling | Columns with 500+ cards will render slowly |
| WebSocket reconnection | No exponential back-off on disconnect |
| Concurrent drag conflict | Last-write-wins; no optimistic lock |
